Configuration management theory

  • Consistency
  • Reproducibility
  • Scalability
  • Version Control

Configuration management allows you to guarantee, to the best of your tools and ability that what you do to one machine, will be done on another. This frees you from having to think about 1000 machines, and instead be able to focus on one machine, then reproduce that state on as many machines as you care to. A side effect of this is that all your machines have a consistent states, and fewer surprises is generally better in the operations world. Another huge advantage is that all your configuration is in SCM, so when you screw something up, or someone wants a configuration back to the way it was 2 weeks ago, the opportunity to step on your own foot when you change it back is minimal. It's interesting how little people on your team will mess around when all their changes are checked into SCM, with their name next to it. Anyone who has written any code probably knows that this won't prevent all kinds of idiocy, but there is significantly less "I don't know who did that and why" floating around.

at my last job we tried this with the router configs ONLY. No changes could be pushed without checking them in first, and despite the fact that there were only 3 people who could read configs, this cut down on around 80% of the nonsense we were all tolerating from one another.

The trifecta

The administration of almost all services in UNIX boils down to 3 things that are dependent on one another.

  • Package
  • Config
  • Service

If you only get this, you'll get a lot. Think about any service you need running, and try and see if it breaks this model. The service depends on the config, the config depends on the package. With puppet, you can leverage this into gospel.

Puppet master configuration

/etc/puppet/
 puppet.conf
 manifests
  nodes.pp
  site.pp
 modules
  hosts
   manifests
    init.pp
   templates
     hosts.erb

This is an /etc/puppet dir. Pay attention to the modules dir, as that's where your library of tools lives. The rest of the files, except for puppet.conf, deal with telling your puppetmaster which nodes need what.

Keep in mind this is only one way of managing the nodes, you can keep them in a database if you please, or have whatever directory structure you want. What I do is I keep a minimal manifest that contains stuff that every server, no matter what it does should have, like ntp, an /etc/resolv.conf, a hosts file that makes sense. that kind of stuff, and then every node inherits this, and the individual stuff for each server goes in it's node definition.

Attaching puppet clients to the master

Your puppet master needs to authenticate all of it's clients using ssl certs. On the client, run 

puppetd --verbose --no-daemonize --server=${master.ip.address}

So at this point we need to try and get our client's cert onto our master. Puppet's default behavior is to drop a request onto the master of each clients that requests a signature, then expects someone to sign each one manually.
