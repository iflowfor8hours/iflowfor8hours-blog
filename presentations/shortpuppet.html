What is it?

Puppet is a configuration management tool that for sysadmins, kinda makes you like IRON MAN. It allows you to weild the power of a team of competent sysadmins that all agree on the way things should be done, despite the fact that there is more than one way to everything. Puppet also allows you maintain a consistent state across all your machines, which leads to fewer surprises. One of my favorite features of puppet is the reporting. The reports that your nodes send back are basically live documentation of your network. Another huge advantage is that all your configuration is in SCM, so when you screw something up, or someone wants a configuration back to the way it was 2 weeks ago, the opportunity to step on your own foot when you change it back is minimal. Trail of blame, this I like.

What is it replacing?

Just as the best companies have realized that outsourcing development is a bad idea because great developers able to build great applications is core to almost every business these days, organizations are realizing that sysadmin productivity should be a primary driver, not automating the people away. The previous generation of operations tools was built around rigid automation that attempted to protect the network from the people, assuming that the most critical work can be done by software. Modern DevOps tools recognize that the best people need powerful tools, not restrictive tools - they want their tools to make them more powerful, like Iron Man, not limited use robots. Because they're powerful these tools don't provide the same kind of protections that more limited tools do, but that same power provides very fast recovery abilities when things do go wrong. To extend the superhero metaphor, with great power comes great responsibility. You can destroy the universe, but without that ability you also can't build the universe from scratch whenever needed.

What could go wrong?

  • TIMTOWTDI
  • Puppet labs has recently shifted their efforts towards marketing an enterprise version of puppet, but it's still open source for the most part. We're not worried about that.

Puppet also allows you to do things in a multitude of ways, and things are still being standardized, so there's no one right way that is optimal. This makes working with others and integrating plugins and modules that people made difficult at times. You can't just pick up someone else's apache setup and shoehorn it into your infrastructure. Puppet was written in ruby from the ground up. Some people don't mind learning ruby. Many people do. In truth the only part that you need ruby for are the templates, which we will see in a second are very simple. You will also need to learn ruby if you want to make plugins, but it's fairly straightforward. Almost everything we need is built-in or within easy reach. Many people bring up a good point which is "ruby is so young and volatile, using it to manage your systems is crazy". This is true to some extent, that ruby changes all the time, but as long as you're not dying for new language features, there's really no harm in using a known good subset of the language.

Intro

Puppet is, in the end a simple client server model. It can be either push or pull, all connections are encrypted and authenticated using SSL. Changes to your environment are pushed to the puppetmaster, and the master then distributes and enforces configuration on your hosts.

Puppet master configuration

/etc/puppet/
 puppet.conf
 manifests
  nodes
   srfarm.net.pp
   vo.srfarm.net.pp
  site.pp
 modules
  apache2
   manifests
    apache2.pp
    init.pp
   templates

This is an /etc/puppet dir. Pay attention to the modules dir, as that's where your library of tools lives. The rest of the files, except for puppet.conf, deal with telling your puppetmaster which nodes need what.

Keep in mind this is only one way of managing the nodes, you can keep them in a database if you please, or have whatever directory structure you want. What I do is I keep a minimal manifest that contains stuff that every server, no matter what it does should have, like ntp, an /etc/resolv.conf, a hosts file that makes sense. that kind of stuff, and then every node inherits this, and the individual stuff for each server goes in it's node definition.

Easy problem 2

SSH keys and openssh-server configuration

I want to make sure ssh is installed, and all my keys are on all my servers, I want to be able to remove or change a key in one place and have it propogated everywhere.

  • How do we address these problems when thinking configuration management (declarative vs imperative)?

The almost universal answer to this question is you have 3 main resources to manage, the installation of whatever package it is, the config file, and the service itself. This is known in the puppet documentation as the trifecta. They say "If you only get this, you'll still get alot". They are correct.

  • How do we make sure everything is done in the right order?

It would be useless to have the config file installed if the package installation failed, or the service called to start if the config file was wrong. The tools puppet gives us to avoid this are notify and require, as shown below.

Class attributes

Important attributes

require => Class["ssh::install"], # ATTENTION PLEASE!!
notify  => Class["ssh::service"],  # WHAT'S THIS???

What do they do?

  • notify will restart the ssh::service anytime the file described inside of ssh::config is modified.

  • require will fail to install the config if ssh::install doesn't return sucessfully.

What about the keys?

One way of doing this is just managing the key file directly. This is almost certainly not the BEST way of doing this, but we'll get into that later. For now, we can just add a definition to our ssh module that depends on the rest of the ssh components, like so

1 class ssh::keys {
2   file { "/root/.ssh/authorized_keys":
3     ensure = > present,
4     owner => 'root',
5     group => 'root',
6     mode => 0600,
7     source => "puppet:///modules/ssh/authorized_keys",
8     require => Class["ssh::install"],
9 }

I took off the notify beacause we don't need to restart ssh every time the keys change. Now, whenever you modify the file that your puppet server is dishing out, it will propogate the keys to all the hosts with the ssh module included.

Ok, so that was pretty easy, and it will save you a bunch of time the first time you use it if you have any more than a couple of servers to deal with.

Everything in facter is accessible by puppet. This becomes a serious amount of leverage when you start using facts to group servers, like webservers in the USA, or timezones, or anything really. We're going to create a custom fact that describes the server's role. Leveraging this with mcollective is awesome. 

        !rb
        # role.rb
        Facter.add("host_role") do
            setcode do
                %x{cat /etc/role}.chomp
            end
        end

This is all there is to a custom fact. This is now available everywhere in puppet and mcollective. We can say all nodes with fact role=db do something.

mCollective

Mcollective really deserves it's own presentation. It's a framework for doing many things with puppet, the most important ones are scheduling, queuing, and rpc-ing. Mcollective allows you to get near instant feedback from your puppet nodes, which in turn allows you to automate things far better than with reports and crontabbed puppet runs. Mcollective also allows you to do node discovery, which makes cloud guys wet and weak in the knees. I use it mostly for kicking all my nodes at once, or all the nodes of a specific subset. If I roll out a change to my web application and have checked it into my repo, I want to see it now. I don't want to wait for the next puppet run. I use mcollective to make a query that will first take all nodes of type application server, and then execute a puppet run on them.

Reporting

Puppet reports are really handy for figuring out what went wrong with each node specifically. They are a report of what occured during the run. I use them constantly as one of my many health gauges of my environment. Enabling them is trivial, it's one line in your node's puppet.conf, and if you're not managing thousands of servers, you can peek at them to see how things are going without ever touching the machine, which I believe is the goal.

Scaling

If you were to go out and just apt-get install puppet and get running with it, you would very soon hit a wall. Puppet ships with a built in webserver called WEBBrick. It's ruby's equavalent of SimpleHTTPServer. Really not for doing anything other than testing. To run puppet in a large environment there are a few options. Most people use passenger, which is mod_rails for apache. Some people use mongrel. I've heard mongrel is really easy to setup, one line or something, but I really don't know anything about it. Apache works fine, I'm sticking with it. There are a ton of different ways to scale puppet. Master replication, splitting your nodes into groups served by individual masters, brokering and queuing connections using a load balancer...
